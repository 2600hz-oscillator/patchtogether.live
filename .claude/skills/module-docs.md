# Module docs: interactive virtual module (primary) + numbered face (fallback)

Every AUDIO module's doc page (`/docs/modules/<id>`) is a **2-column explorer**:
- **LEFT** — the **interactive virtual module**: the REAL card component (mounted
  client-only in a throwaway sandbox rackspace), which you can hover and open
  patch panels on. Because it renders the real card, faceplate changes show up
  automatically — no screenshot to regenerate.
- **RIGHT** — a **hover pane** (`DocHoverPane`) that explains whatever faceplate
  control OR patch port you hover, from the AUTHORED `docs`. A CV input that
  modulates a param shows the DUAL context ("modulates Attack — <what the Attack
  fader does>"). Empty state = the module explanation.

The card renders live; the prose is authored + drift-gated. The **numbered card
screenshot + authored KEY** is now the **no-JS / prerender / not-yet-promoted
FALLBACK** (the static face the prerendered HTML carries so the page is readable
without JS, and the view for modules not yet on the live-card allowlist). NEVER a
raw-test-id legend — that was explicitly rejected.

Interactive pieces (prototype on adsr + sequencer): `$lib/docs/doc-index.ts`
(`buildDocIndex` — the flat, prerender-safe payload the pane indexes),
`$lib/docs/interactive/{VirtualModule,DocHoverPane}.svelte` +
`use-doc-hover.svelte.ts` (the hover action: resolves a hovered element to a port
via `data-port-id`+`data-direction`, else a control via `control-<id>` /
control-family / button testid, reusing `control-doc-resolver`), and
`interactive-doc-modules.ts` (the `INTERACTIVE_DOC_MODULES` allowlist — promote a
module here once its live card is vetted in the doc sandbox; a module can be
STRICT-documented yet not yet on this list). Sandbox isolation: VirtualModule
`bindRackspace`es a throwaway LOCAL Y.Doc (bindRackspace never attaches the
relay — only the rackspace page does), seeds one stub node, provides a null
engine, and tears the binding down on unmount, so a doc card can NEVER mutate a
real rack or open a multiplayer room. e2e: `tests/docs-virtual-module.spec.ts`.

See also: `running-tests`, `module-development`, `vrt-failures`. Design memory:
`docs-numbered-key-system`.

## The pieces (don't reinvent)

- **Authored prose** lives co-located on the def: `docs: { explanation, inputs:
  {<id>:str}, outputs: {<id>:str}, controls: {<key>:str} }` (`ModuleDocs` in
  `$lib/graph/types`). `inputs`/`outputs` are split because a module can have a
  same-id input AND output (e.g. sequencer `clock`). Model it like
  `lib/audio/modules/sequencer.ts`.
- **controls keys** resolve three ways (`$lib/docs/control-doc-resolver.ts`):
  1. a **param** id (`bpm`, `attack`, …) — the Knob/Fader `control-<id>` testid;
  2. a **family template** `<prefix>-{n}` — a repeated grid (`seq-gate-{n}`);
     write ONE entry, `{n}` is substituted per member / shown as ×N when the
     face collapses the family;
  3. a **static button** key = its test id with the runtime nodeId stripped
     (`sequencer-play`, `quicksave-mode-save`). Anonymous buttons (`button:N` in
     the legend) need a real `data-testid` on the card first.
- **Numbered face + legend** are generated by `e2e/vrt/vrt-annotated.spec.ts`
  (overlay = `e2e/vrt/annotate-controls.ts`) → `__annotated__/darwin/<t>.png` +
  `<t>.legend.json` (number → stable testid, nodeId normalized to `{id}`). Dense
  control families collapse to ONE numbered callout (≥3 members) so a 16-step
  grid is one number, not 16 — this is what keeps the face uncluttered.
- **Drift gate**: `contract-lock.test.ts` (def contract) + `module-docs-lint.test.ts`
  (consistency + STRICT completeness: every numbered control on a promoted
  module must resolve to non-empty prose). `strict-docs.ts` is the promoted set.

## Author a module (the per-module loop)

1. **Read the truth**: the def (`lib/audio/modules/<t>.ts`), its card
   (`lib/ui/modules/*Card.svelte` + shared sub-components), and project memory
   for *why* the module exists / how controls behave. Docs must be behavioral
   ("Gate 3 turns step 3's gate on/off"), not mechanical.
2. **Write `docs`** on the def: a model-level `explanation`, then EVERY input,
   output, param, control family, and static button. Trigger/gate port prose
   must use its declared `edge` vocabulary (the lint checks this).
3. **Promote**: add the type to `STRICT_DOCS` in `strict-docs.ts`.
4. **Generate the face**: `flox activate -- task e2e:serve` then
   `ANNOTATED_MODULES="<t>" flox activate -- task vrt:annotated` (writes the
   numbered PNG + legend + copies to static). `REPEAT=3 task vrt:annotated:one --
   <t>` to flake-check.
5. **Re-pin goldens**: `flox activate -- task docs:accept` (regenerates
   `contract-lock.txt` + `module-docs.generated.ts`).
6. **Verify**: `flox activate -- task typecheck` +
   `flox activate -- task test:one -- src/lib/docs/` (lint completeness will name
   any numbered control with no authored blob; the doc-index test covers the flat
   payload + CV→param dual link) + `task e2e:one -- tests/docs.spec.ts`. When the
   module is on the live-card allowlist, also run
   `task e2e:one -- tests/docs-virtual-module.spec.ts`.
7. **EYEBALL the face PNG** (`__annotated__/darwin/<t>.png`). Numbers must not
   obscure controls — see Layout below.

## Layout: numbers must not cover controls OR label text

`annotate-controls.ts` places each numbered circle with a leader line back to a
dot on the control. The rules (all already implemented — tune, don't rebuild):

- **Collapse dense families**: a repeated grid (≥3 controls sharing a
  `<prefix>-<nodeId>-<i>` test id) becomes ONE callout (the legend entry carries
  `count`). Keeps a 16-step grid from spraying 16 circles.
- **Never sit on another CONTROL or circle** (`hitsControl`) — those are hard
  blockers.
- **Never cover a control's NAME LABEL / value text** (`hitsLabel`, selector
  `LABEL_OBSTACLE_SELECTOR` = `[class~="label"], .value-tag`). A Knob/Fader
  renders its label (ATTACK/DELAY/…) right below the control, so the circle
  would land on it. Instead the placement **slides DOWN past the label** into
  clear space (the leader line lengthens, the label stays readable). It only
  flips UP when a real control blocks below (e.g. a dense grid's top row tucks
  into the gap above instead of running a long leader across the card).

If a face still looks crowded after regenerating:
- A control rendering with no `data-testid` shows as `button:N` and can't be
  keyed — add a stable `data-testid={`<name>-${id}`}` to the card element.
- A label that's still covered probably uses a class the selector misses — widen
  `LABEL_OBSTACLE_SELECTOR`.
- A family that didn't collapse has test ids not matching `<prefix>-<id>-<i>` —
  fix the card's test ids.
- Last resort: tune `R` / `LEADER` / `PAD` / the down-then-up search in
  `annotate-controls.ts`. ALWAYS eyeball the regenerated PNG
  (`__annotated__/darwin/<t>.png`) — this is a visual loop, not a unit assertion.

## Rollout to all modules

Batch fully (~10 modules/batch: author + face + promote + all gates green before
the next), authored via agent fan-out (one agent per module from code + memory,
adversarially checked; verify the gate stays green). Each batch is its own PR off
`main`. Video modules have no `[id]` page — audio only. (task #63.)

## docs are hash-transparent to attest

Authoring docs must NOT churn the webgl/collab attest hashes. Audio defs aren't
in those bases; video defs wrap their `docs` in `// docs-hash-ignore:start/end`
markers (`stripDocsForHash`). If a docs edit ever flips an attest, that's a bug —
see `docs-numbered-key-system`.
